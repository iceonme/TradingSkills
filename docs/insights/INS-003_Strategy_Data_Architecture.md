# INS-003: Strategy Skill 数据流逝架构取舍 (Active vs Passive Data Fetching)

**日期**: 2026-02-28
**主题**: 策略微服务 (`grid_rsi_4.0_Jeff_Huang`) 是应该“被动接收数据 (Push/喂入)” 还是“主动拉取数据 (Pull/自生自灭)”？
**决策状态**: [草案/讨论中] 待次日确认。

## 1. 核心矛盾
在设计标准化的交易策略 Skill 时，面临数据流向的选择：
- **方案 A (Push 模式 / Pure Function)**：微服务内部完全失明。它暴露一个 `analyze(kline_window)` 函数，由外围中枢（如 OpenClaw PA 或 TAA 回测引擎）按照一定的节奏定期把 K 线数据塞进去，它只负责吐出动作结论。
- **方案 B (Pull 模式 / Autonomous Microservice)**：微服务自己掌控节奏。它内部集成并调用数据源接口（如 `OKXAPI()`），外部只需发送“开始运行”指令，它就会在内部自循环获取数据并计算。

## 2. 架构考量与痛点分析

### 为什么界内几乎 100% 倾向于 Push (方案 A) ？
1. **回测一致性 (Backtest Parity) 致命缺陷**：
   如果采用 Pull 模式，策略在回测中会试图去向外网发起请求。为了跑历史回测，必须在最底层编写极其复杂的网络请求拦截与时间线 Mock 系统。
   相反，Push 模式是天生的“纯函数”，给啥定点历史数据出啥结论，回测引擎只需用极快的 `while` 循环不断硬塞数据即可，完美的“一份代码，双端运行”。
2. **连接池雪崩 (Connection Avalanche)**：
   如果多个 Agent 挂载了多个策略微服务同时各自去拉盘口数据，会导致 API 并发极速超限被封。统一由 PA 代理人拉取一次后广播分发 (Push) 是工程效能的必然。
3. **黑盒化不可诊断**：
   PA 作为高层调度者，如果不知道策略到底“看了哪门子的数据”就瞎发出了买单，出事故后无法归因。由 PA 亲自递送的数据快照可作为最佳的追溯日志。

## 3. 面向 Agent 的折衷提议 (Adapter Pattern 适配器模式)
为了兼收“Agent 调用免操心”与“底层算法可完美回测”的双重红利，提议采用**“智能外衣 + 纯洁脑干”**的内外层隔离设计：

- **内核 (`strategy/core.py`)：绝对的 Push 模式。**
  严格遵守状态剥离协议。这里的核心逻辑算法永远是瞎子，只能处理传进来的 `window` 字典数组，确保在 TAA 引擎里跑得飞起。
- **外壳 (`scripts/run_live_agent.py`)：实现 Pull 模式。**
  作为专门向 OpenClaw 这种糊涂虫暴露的“保姆式外衣”。这层脚本内部自己起一个定时器，自己去调用我们在别的 Skill (如 `okx_trading_exe`) 里的能力拿数据，然后自动调用体内的 `core.py` 喂入数据。
